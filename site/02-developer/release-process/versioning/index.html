Versioning Policy¶

Semantic versioning policy for Sellia releases.

Overview¶

Sellia follows Semantic Versioning 2.0.0.

Format: MAJOR.MINOR.PATCH (e.g., 1.2.3)

  • MAJOR: Incompatible API changes
  • MINOR: New functionality (backwards compatible)
  • PATCH: Bug fixes (backwards compatible)

Version Number Format¶

Examples¶

Version Type Meaning
0.1.0 Initial First release
0.4.0 Pre-release Current development version
1.0.0 Major Future stable release

Pre-Release Versions¶

Format: MAJOR.MINOR.PATCH-prerelease.N

Examples: - 1.0.0-alpha.1 - 1.0.0-beta.1 - 1.0.0-rc.1

Version Bumping Rules¶

MAJOR (X.0.0)¶

Increment when:

  1. CLI interface changes
  2. Removing commands or flags
  3. Changing flag names or formats

  4. Requiring new arguments

  5. Config file format changes

  6. Removing config fields
  7. Renaming required fields

  8. Changing config file structure

  9. Protocol changes

  10. Breaking protocol message changes
  11. Removing message types

  12. Required new fields in messages

  13. API key changes

  14. Requiring new API key format
  15. Changing authentication method

Examples: 

1.2.3  2.0.0
- Remove `--legacy-flag` option
- Change config from YAML to TOML
- Require API keys for all tunnels

MINOR (x.Y.0)¶

Increment when:

  1. New features
  2. Adding new commands
  3. Adding new CLI flags (optional)

  4. Adding new config fields (optional)

  5. Enhancements

  6. New inspector features
  7. Performance improvements

  8. New protocol message types (backwards compatible)

  9. Non-breaking additions

  10. New tunnel types
  11. New routing features
  12. Additional output formats

Examples: 

1.2.3  1.3.0
- Add `sellia admin` command
- Add WebSocket inspection in inspector
- Add path-based routing

PATCH (x.y.Z)¶

Increment when:

  1. Bug fixes
  2. Fix crashes
  3. Fix memory leaks

  4. Fix incorrect behavior

  5. Internal changes

  6. Code refactoring
  7. Performance optimizations

  8. Documentation updates

  9. Backwards-compatible fixes

  10. Improve error messages
  11. Add validation warnings
  12. Fix edge cases

Examples: 

1.2.3  1.2.4
- Fix connection timeout issue
- Fix inspector display bug
- Improve error message for invalid auth

Pre-Release Versions¶

Alpha¶

Status: Early development, unstable

Usage: Feature development, testing

Examples: - 1.0.0-alpha.1 - First alpha - 1.0.0-alpha.2 - Second alpha

Beta¶

Status: Feature complete, testing needed

Usage: Public testing, feedback

Examples: - 1.0.0-beta.1 - First beta - 1.0.0-beta.2 - Second beta

Release Candidate (RC)¶

Status: Ready for release unless bugs found

Usage: Final testing before GA

Examples: - 1.0.0-rc.1 - First RC - 1.0.0-rc.2 - Second RC

Development Versions¶

Nightly / Dev Builds¶

Format: dev-YYYYMMDD or commit hash

Examples: - dev-20241230 - dev-a1b2c3d

Status: Unstable, not for production use

Version Sources¶

Source of Truth¶

  1. src/core/version.cr - Crystal code (primary) 

    module Sellia
  VERSION = "0.4.0"
end

  2. shard.yml - Crystal dependencies 

    name: sellia
version: 0.4.0

  3. package.json - Web UI (if UI changed) 

    {
  "version": "0.4.0"
}

  4. Git tags - Releases 

    v0.4.0

They must be synchronized for releases.

Release Workflow¶

1. Development¶

# Work on main branch
git checkout main

# Make changes, commit
git commit -m "feat: add new feature"

Version format: dev or unreleased

2. Pre-Release¶

# Update version in src/core/version.cr
VERSION = "1.3.0-alpha.1"

# Commit
git commit -m "chore: bump version to 1.3.0-alpha.1"

# Tag
git tag v1.3.0-alpha.1
git push origin v1.3.0-alpha.1

3. Release¶

# Update version in src/core/version.cr
VERSION = "0.5.0"

# Update shard.yml
# version: 0.5.0

# Update package.json (if UI changed)
npm version 0.5.0

# Commit
git commit -am "chore: release v0.5.0"

# Tag
git tag v0.5.0
git push origin main v0.5.0

# GitHub Actions creates release and builds binaries

Backwards Compatibility Policy¶

Guaranteed Stable¶

These interfaces are stable within a MAJOR version:

  1. CLI Commands
  2. Existing command names
  3. Existing flag names

  4. Flag formats

  5. Config File

  6. Existing config keys

  7. Config file format (YAML)

  8. Protocol

  9. Existing message types

  10. Message field names

  11. Public URLs

  12. URL format for tunnels

May Change¶

These interfaces may change in MINOR releases:

  1. Output format
  2. CLI text output

  3. Log messages

  4. Internal APIs

  5. Database schema

  6. Inspector API (not public)

  7. Default values

  8. Port numbers
  9. Timeout durations

Deprecation Policy¶

Deprecation Process¶

  1. Announce deprecation in release notes (MINOR version)
  2. Mark as deprecated in code/docs
  3. Wait one MAJOR version before removal
  4. Remove in next MAJOR version

Example: 

v1.2.0 - Add new feature
v1.3.0 - Mark old feature as deprecated (release notes)
v2.0.0 - Remove old feature

Changelog Requirements¶

For Each Release¶

Must include:

  • Version number and date
  • Type (MAJOR/MINOR/PATCH)
  • Changes grouped by type:
  • Added
  • Changed
  • Deprecated
  • Removed
  • Fixed
  • Security

See: Changelog Maintenance

Migration Guides¶

For MAJOR Releases¶

Provide migration guide when:

  1. Config format changes
  2. CLI flags change
  3. Protocol changes

Example: 

## Migrating from 1.x to 2.0

### Config Changes

Old format:
```yaml
tunnel_port: 3000

New format: 

tunnels:
  default:
    port: 3000

---

## Version Detection

### Check Version

```bash
sellia version
# Output: Sellia v0.4.0

sellia-server --version
# Output: Sellia Server v0.4.0

Check for Updates¶

sellia update --check
# Output: Current: 1.2.3, Latest: 1.2.4
