Release Process¶

How Sellia releases are managed and published.

Overview¶

This section covers the release process for Sellia, including versioning, changelog generation, and publishing releases.

Versioning¶

Sellia uses Semantic Versioning:

  • MAJOR - Incompatible API changes
  • MINOR - Backwards-compatible functionality
  • PATCH - Backwards-compatible bug fixes

Format: MAJOR.MINOR.PATCH

Examples: - 1.0.0 1.0.1 (Bug fix) - 1.0.0 1.1.0 (New feature) - 1.0.0 2.0.0 (Breaking change)

Release Checklist¶

Pre-Release¶

  1. All tests passing 

    crystal spec

  2. Documentation updated

  3. Update README if needed
  4. Update docs/ for new features

  5. Update CHANGELOG.md

  6. Version bumped

  7. Update version in src/core/version.cr
  8. Update version in shard.yml
  9. Update version in web/package.json (if UI changed)

Creating Release¶

  1. Create release branch 

    git checkout -b release/v1.0.0

  2. Update version files

  3. src/core/version.cr: VERSION = "0.4.0"
  4. shard.yml: version: 0.4.0

  5. web/package.json: version: "0.4.0" (if UI changed)

  6. Update CHANGELOG.md 

    ## [1.0.0] - 2025-01-30

### Added
- New feature X
- New feature Y

### Fixed
- Bug fix Z

### Changed
- Updated dependency A

  7. Commit changes 

    git add src/core/version.cr shard.yml web/package.json CHANGELOG.md
git commit -m "chore: release v0.4.0"

  8. Build and test 

    cd web && npm ci && npm run build && cd ..
shards build sellia --release -Dembed_assets
shards build sellia-server --release --static
crystal spec

  9. Merge to main 

    git checkout main
git merge release/v1.0.0

  10. Tag release 

    git tag -a v0.4.0 -m "Release v0.4.0"
git push origin main
git push origin v0.4.0

Post-Release¶

  1. Create GitHub Release
  2. Go to GitHub releases page
  3. Click "Draft a new release"
  4. Choose tag v0.4.0
  5. Copy changelog from CHANGELOG.md
  6. GitHub Actions will automatically build and publish binaries and Docker images

  7. Publish release

  8. Build release binaries (future) 

    # Build for multiple platforms
# Upload to GitHub release

  9. Announce release

  10. Update documentation
  11. Post announcement
  12. Update website

Changelog Format¶

Sellia uses Keep a Changelog format:

# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.0.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added
- Upcoming features

### Changed
- Upcoming changes

## [1.0.0] - 2025-01-30

### Added
- Initial release
- HTTP tunnel support
- Request inspector
- Basic authentication
- Subdomain routing

### Fixed
- Connection stability issues

## [0.1.0] - 2025-01-15

### Added
- First beta release

Release Branch Strategy¶

Main Branch¶

  • Active development and stable releases
  • Tags for each release
  • Production-ready code

Feature Branches¶

  • New features
  • Bug fixes
  • Merged to main via pull requests

Hotfix Process¶

For urgent fixes:

  1. Create hotfix branch 

    git checkout -b hotfix/v0.4.1

  2. Implement fix 

    # Make changes
git commit -m "fix: critical bug"

  3. Build and test 

    crystal spec

  4. Merge to main 

    git checkout main
git merge hotfix/v0.4.1

  5. Tag and release 

    git tag -a v0.4.1 -m "Hotfix v0.4.1"
git push origin main --tags

Dependency Updates¶

Crystal Dependencies¶

Update shard.yml:

dependencies:
  message-pack:
    github: crystal-community/message-pack-crystal
    version: ~> 1.0.0  # Update version

Run update:

shards update

Node Dependencies¶

Update web/package.json:

{
  "dependencies": {
    "react": "^18.2.0"
  }
}

Run update:

cd web
npm update

Testing Before Release¶

Full Test Suite¶

# Crystal tests
crystal spec

# UI tests
cd web
npm test
cd ..

# Integration tests
./scripts/integration-test.sh

Manual Testing¶

  1. Server startup 

    ./bin/sellia-server --port 3000 --domain localhost

  2. Tunnel creation 

    ./bin/sellia http 3000

  3. Version check 

    ./bin/sellia version
./bin/sellia-server --version

  4. Request handling

  5. Make requests through tunnel
  6. Verify responses

  7. Check inspector

  8. Configuration

  9. Test config file loading
  10. Verify environment variables
  11. Check CLI flags

Platform Testing¶

Test on supported platforms: - Linux (Ubuntu, Debian, CentOS) - macOS (Intel, Apple Silicon) - Docker

Version Bumping Examples¶

Patch Release (Bug Fix)¶

# 1.0.0  1.0.1
git checkout -b release/v1.0.1
# Update shard.yml: version: 1.0.1
git commit -m "chore: release v1.0.1"
git tag -a v1.0.1 -m "Release v1.0.1"

Minor Release (New Feature)¶

# 1.0.0  1.1.0
git checkout -b release/v1.1.0
# Update shard.yml: version: 1.1.0
git commit -m "chore: release v1.1.0"
git tag -a v1.1.0 -m "Release v1.1.0"

Major Release (Breaking Change)¶

# 1.0.0  2.0.0
git checkout -b release/v2.0.0
# Update shard.yml: version: 2.0.0
git commit -m "chore: release v2.0.0"
git tag -a v2.0.0 -m "Release v2.0.0"
